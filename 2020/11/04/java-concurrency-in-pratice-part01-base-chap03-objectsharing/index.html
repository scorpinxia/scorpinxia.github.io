JAVA Concurrency
发布日期:   2020-11-04

可见性

多个线程读写

需要保证多个线程对内存写入同步


  • 问题

    • 可能无限循环，读线程可能永远看不到ready值
    • 可能输出0，看到了写入ready 的值确没看到写入number的值 “重排序”现象
    • 主线程首先写入number 没有同步的情况到写入ready

  • Tips

    • 没有同步的情况下，编译器，处理器可能堆操作执行顺序进行意想不到的调整

失效数据

调用get 任然可能看到失效值

非原子的64位操作

最低安全性：线程没同步读取，读取

对非volatile的LONG 和double 变量，JVM允许64bit的读写分为两个32bit的操作。读写在不同线程，可能读到某个值的高32bit和另一个的低32bit

加锁和可见性

为什么所以线程同一个锁同步，确保B执行锁保护代码时，可以看到线程A之前在同一个同步代码快的所有操作

加锁的意义

不仅仅局限在互斥，还有内存可见性。所有读操作和写操作得到线程必须在同一个锁上同步

Volatile

弱同步，确保变量更新通知到其他线程。

效果

  • 编译器和运行时注意这个变量是共享的，不会和其他内存重排序，不会被缓存在寄存器或者其他处理器不可见的地方。
  • 读取总会返回最新写入的值

使用领域

  • 确保状态的可见性
  • 确保它们引用变量状态的可见性
  • 标识 重要程序什么周期事件发生（初始化，关闭）

注意点

  • 不足以确保递增原子性conut++
  • 加锁可以确保，可见性和原子性，volatile只能确保可见性

需要满足的条件

  • 对变量的写入操作不依赖变量当前值，或者确保单个线程更新变量值
  • 变量不和其他变量一起纳入不变性条件
  • 访问变量时不需要锁

发布(publish)和逸出(escape)

  • 发布(publish): 使对象能够在当前作用域之外代码使用。例子，发布到公有的静态变量

  • 逸出(escape)：某个不该发布的对象被发布

    本应私有的变量被发布了（可以修改）

隐含的发布了ThisEscape实例本身（内部类实例包含对ThisEscape实例的隐含引用

安全的对象构造

this引用不能在构造过程escape

常见错误：在构造函数注册一个事件监听、或者启动线程

解决例子，用私有构造加上工厂方法（factory Method）：只在构造函数返回的时候从线程escape this引用

线程封闭 （Thread confinement）

不共享数据，在单线程内访问数据

JDBC(Java Database Connectivity) 的Connection对象

Java语言无法强制将对象封闭到某个线程，只有一些机制

ad-hoc线程封闭

完全由程序实现，比较难

例子：单个线程对共享的volatile写入，相当于把修改操作封闭在单个线程上

不推荐使用

栈封闭

只能通过局部变量才能访问对象

  • 基本类型的局部变量，无论如何都不会破坏栈封闭
  • 实例化一个TreeSet对象，对象引用保存到animals,被封闭在局部变量里面

ThreadLocal

最规范的方法:这个类使线程中某个值与保存值的对象关联。为每个使用该变量的线程存有独立的副本。get总是返回当前执行线程set的最新值

通过将JDBC连接保存到ThreadLocal对象，每个线程有自己的连接

某个频繁执行的操作需要临时对象（缓冲区），又希望避免每次执行重新分配改临时变量

概念上ThreadLocal 看成包含了Map<Thread,T>对象

不变性

不可变对象一定线程安全

满足条件

  • 对象创建后，状态不可变
  • 对象的所有域是final类型
  • 对象正确创建 （this引用无escape)

不可变的对象和不可变的对象引用有差别

final

C++ const的受限版本

  • 除非需要 所有域 都应该私有
  • 除非需要变化 final

Volatile发布不可变对象

对相关数据原子操作时，可考虑用不可变类包含这些数据

使用指向不可变容器对象的Volatile类型缓存最新的结果

安全发布

不正确的发布

可能看到尚未创建完成的对象

其他线程看到的Holder 对象可能是失效值。 某线程第一次读取域读到失效值，第二次读取域得到更新值 就throw Assertion

不可变对象和初始化安全性

final 发布不可变对象保证初始化安全性

  • 如果final 类型域指向可变对象，访问这些域指向对象的状态还是需要同步

安全发布常用

为了安全地发布对象，对对象的引用和对象的状态必须同时对其他线程可见。正确构造的对象可以通过以下任意方式安全发布

  • 从静态初始化器初始化对象引用；
  • 将对对象的引用保存到a volatile field or AtomicReference 中；
  • 将对对象的引用保存到正确构造的对象的final类型域中；
  • 将对对象的引用存储到由锁适当保护的域中。

线程安全容器，是通过最后一个条件

  • 将键或值放入a Hashtable, synchronizedMap, o r Concurrent-Map中，安全地将其发布到从映射中检索它的任何线程(无论是直接还是通过迭代器)；
  • 将元素放在Vector, CopyOnWriteArrayList,CopyOnWrite-
    ArraySet, synchronizedList, or synchronizedSet，安全地将其发布到从集合中检索它的任何线程；
  • 将元素放在a BlockingQueue or a ConcurrentLinkedQueue上，可以安全地将其发布给从队列中检索它的任何线程。
  • 类库中的其他切换机制(如Future和Exchanger)也构成安全发布；

使用静态初始化器通常是发布可以静态构造的对象最简单最安全的方法:公共静态Holder Holder = new Holder(42)；

静态初始化器由JVM在类初始化时执行；由于JVM中的内部同步，这种机制可以保证安全地发布以这种方式初始化的任何对象。

事实不可变对象

对象技术可变，但是状态发布后不会改变
Effectively Immutable Obeject

比如Date 本身可变，可以当做不可变对象使用

如果Date对象值放入Map之后不会改变，synchronizedMap的同步机制就够让Date值安全发布

安全地共享对象（策略）

每当你获得一个对象的引用时，你应该知道你可以用它做什么。使用前需要获取锁吗？是允许修改它的状态，还是只允许读？许多并发性错误源于未能理解共享对象的这些“参与规则”。发布对象时，应该记录如何访问该对象。

在并发程序中使用和共享对象最有用的策略是:

  • 线程封闭。一个线程限制的对象被一个线程独占和限制，并且可以被它自己的线程修改。
  • 共享只读。共享只读对象可以由多个线程并发访问，无需额外的同步，但不能由任何线程修改。共享只读对象包括不可变的和事实不可变的对象。
  • 线程安全共享。线程安全对象在内部执行同步，因此多个线程可以通过其公共接口自由访问它，而无需进一步同步。
  • 保护对象。只有持有特定的锁，才能访问受保护的对象。受保护对象包括封装在其他线程安全对象中的对象，以及已知由特定锁保护的已发布对象。
