同步容器类的问题

某些复合操作需要客户端额外的加锁，迭代遍历，跳转、条件运算。

虽然没有加锁，复合操作任然线程安全，但是多线程并发修改的时候可能出现意料之外的行为

迭代器与ConcurrentModificationException

解决方法的权衡

  • 向之前那样加锁，在dosomething时持有锁，可能导致死锁（dosomthing也需要 list的锁）。
  • 就算没有死锁问题，持有锁时间过长导致很多线程竞争锁，吞吐量和CPU利用率。
  • “克隆”容器迭代，确定有显著的性能开销

容器的hashcode和equals 也会间接执行迭代

并发容器 （并发性重点）

同步容器的缺点

  • 把并发状态访问串行化，吞吐量严重降低

并发容器是目标

  • 针对多个线程并发访问设计的

Java5的并发容器

  • Concurrent-HashMap，替代同步的Map
  • CopyOnWriteArrayList,用于遍历操作为主要操作的情况下代替同步List
  • ConcurrentMap接口增加了常见复合操作的支持，if not add ，替换和有条件删除

新的容器类型

  • Queue 保存一组等待处理的数据，Queue的操作不会阻塞，队列空，返回空值。

  • 虽然可以用List模拟Queue行为–也正是通过LinkedList来实现Queue的，还是需要Queue，去掉List的随机访问需求，更高效并发

    • ConcurrentLinkedQueue:传统的先入先出队列
    • PriorityQueue,非并发的优先队列。

  • BlockingQueue

    • 扩展了Queue 增加了可阻塞的插入和获取，
    • 队列空，获取元素的操作阻塞，知道出现第一个可用的元素；
    • 队列已满，插入一直阻塞知道队列有可用空间 （生产者-消费者设计模式 需求）

java6 并发容器

  • ConcurrentSkipListMap替代同步的SortedMap（用synchronizedMap包装的TreeMap)
  • ConcurrentSkipListSet替代同步的SortedMap（用synchronizedMap包装的TreeSet)

Concurrent-HashMap

粒度更细的加锁机制：分段锁

  • 任意读取线程并发访问Map
  • 执行读操作线程和写入操作线程并发访问MAP，一定数量的写入线程可以并发修改Map
  • 并发环境更高吞吐量，单线程环境损失很少性能
  • 弱一致性的迭代器而不是“及时失败”
    • 不会ConcurrentModificationException ，所以迭代过程不用加锁
    • 容忍并发修改，可以（但不保证）修改后操作反映给容器
    • size、isEmpty可能过期,但是并发环境用处较小
    • 换取到其他重要操作的性能优化，get、put、containsKey remove等

只有独占访问，才应该放弃ConcurrentHashMap 用Hashtable

额外的原子Map操作

ConcurrentMap接口 添加了一些常见的复合操作 也都实现了原子操作

CopyOnWriteArrayList

代替同步的List

  • Copy-on-Write 容器线程安全性，在与只要发布一个事实不可变对象，访问对象就不需要进一步修改。
  • 每次修改创建重新发布一个新的容器副本
  • Copy-on-Write 容器迭代器保留一个指向底层基础数组的引用（不会被修改的，所以多个线程可同时迭代）
  • 每次修改容器复制底层，所以是迭代远大于修改时用

适用于事件通知系统的特征

  • 分发通知时需要迭代调用每个注册的监听器
  • 注册和注销事件监听器操作员少于接受事件通知

阻塞队列 生产者–消费者模式

BlockingQueue的方法

  • 可阻塞的put和take （无界队列 可无限put）
  • 定时的offer和poll （offer 数据不能添加返回失败状态，从而用灵活的方式处理）

支持 生产者和消费者设计模式 分离了找出和执行的过程

  • 找出需要完成的工作
    • 工作放到待完成列表
  • 执行工作
  • 因为两个过程的处理速率不同

BlockingQueue的生产者和消费者

  • 数据生成，生产者把数据放入队列（不需知道消费者的特征）
  • 消费者准备处理数据，从队列中获取（不需知道生产者的特征）

线程池和工作队列的组合（后续）

开发者会假设消费者处理工作速率大于生产者，所以不设置边界。

  • 需要设计中构建资源管理

BlockingQueue的实现

  • LinkedBlockingQueue FIFO队列 先入先出
  • ArrayBlockingQueue FIFO队列
  • PriorityBlockingQueue FIFO用优先级排序 ，而不是FIFO
  • SynchronousQueue 不是真正的队列 ，不为元素维护存储空间，而是维护一组线程等待把元素加入移除队列。
    • 直接交付
    • 没有存储功能，put get一直阻塞等待另一个线程准备好参与交付
    • 消费者足够多 适合同步队列

生产者消费者例子–桌面搜索

代理程序 ，扫描本地驱动器的文件建立索引方便搜索

  • 生产者：找出所有文件 put
  • 消费者：把队列的文件index

多个爬虫和索引建立程序

  • 消费者线程永远不退出，程序无法终止（后续解决）

串行线程封闭

BlockingQueue 确保了一个线程接受转移的对象

双端队列BlockingDeque 和工作密取（Working Stealing

BlockingDeque的目的 工作密取

  • 每个消费者有各自的双端队列，完成了自己双端队列的全部工作，可以从其他消费者双端队列末尾密码获取工作
  • 比传统模式更高的可伸缩行，不会在单个共享任务队列竞争，大多时候访问自己的双端队列。（访问另一个队列的队尾，更降低了竞争）

使用范围

  • 既是消费者也是生产者（某个工作执行会出现更多的工作）
    • 处理一个页面，有更多的页面处理
    • 垃圾回收堆标记 Working Stealing 高效并行
    • 工作线程找到新任务单元，放到自己队列的队尾

阻塞和中断方法

阻塞和暂停执行的原因

  • 等待I/O操作结束
  • 等待获得一个锁
  • 等待从Thread.sleep方法醒来
  • 等待另一个线程的计算结果

阻塞的状态

  • BLOCKED
  • WAITING
  • TIMED_WAITING

与很长执行时间的普通操作的区别

  • 需要外部不受它控制的事件发生才能回到RUNNABLE状态，并再次被调度执行

Interrupted-Exception

  • 表示该方法是阻塞方法，如果被中断，它将努力提取结束阻塞状态
    • BlockingQueue 的put 和take会抛出 （Check Excepiton)Interrputed Exception
    • Thread.sleep
  • Thread提供interrput方法，用于中断线程和查询线程是否已经被中断，每个线程有个boolean 中断状态

中断的机制

  • 协作机制：一个线程不能强制其他线程停止正在执行的操作而执行其他操作
  • A中断B 代表A要求B执行到某个可以暂停的地方—前提是B原因停止（API无特定级别的语义）
  • 最常用的中断就是取消某个操作。

处理抛出Interrupted-Exception方法

  • 传递InterruptedException
    • 根本不捕获该异常
    • 捕获该异常，执行简单的清理工作在抛出这个异常
  • 恢复中断

不应该做的事

  • 捕获它不做任何响应

同步工具类

定义

  • 根据自身状态协调线程的控制流
    • 容器类的 阻塞队列 不仅是保存对象容器，还能协调生产者消费者线程的控制流
    • 信号量 Semaphore
    • 栅栏 Barrier
    • 闭锁 Latch

闭锁

定义

  • 延迟线程进度直到其达到终止状态
  • 相当于一扇门，一直关闭没有任何线程能通过。

作用

  • 确保某个计算需要的资源全部初始化。二元闭锁（两个状态）需要R的都在这个闭锁等待
  • 确保某个服务的依赖服务开启。在启动S服务时，在S依赖的其他服务闭锁等待
  • 等待直到某个操作的所有参与者（多玩家游戏）

闭锁实现

  • CountDownLatch
    • 使一个或多个线程等待一组事件发生
    • 闭锁状态包括一个计数器，初始化为正数，表示需要等待的数量
    • countDown方法递减计数器
    • 计数值非0 await会一直阻塞直到计数值0，或等待中的线程中断，或者等待超时


  • 两个门
    • 开始门：等所有线程创建完毕，开门 计时
    • 结束门，等待所有任务做完
  • 为了测试n个线程并发执行某个任务的实际 （启动门使得主线程能够同时释放所有工作线程） （结束门使得主线程等待最后一个线程执行完成，而不是顺序等待）

FutureTask

定义

  • 也可用作闭锁
  • 表示一种抽象可生成结果的计算
  • 通过Callable实现（可生成结果的Runnable）
  • 三种状态
    • 等待运行
    • 正在运行
    • 运行完成 （表示计算的所有可能：正常结束、异常结束）
  • get 阻塞到任务完成状态返回

例子

  • Preloader 创建一个futureTask 包含从数据库加载产品信息的任务，执行运算的线程

  • 构造函数和静态初始化启动线程不是好方法，（所以另外用了start（））

  • 需要ProductInfo 可以用get 等待加载完成

  • Callable代表任务可以抛出受检查的和未受检查的异常，任何代码可能抛出error。无论任务代码抛出什么异常，都被封装到ExecutionException 并在get中重新抛出

  • ExecutionException的三种情况

    • Callable抛出的受检查异常
    • RuntimeExcepiton
    • Error

  • 首先Preloader会检查已知的受检查异常，并重新抛出他们

  • 然后调用launderThrowable抛出

信号量Semaphore

定义

  • 用来控制同时访问某个特定资源、某个操作的操作数量，资源池，容器边界

  • Semaphore信号量通过虚拟的许可,许可数量可指定

  • acquire 阻塞得到许可

  • release 返回一个许可给信号量

    特殊情况
    二值信号量，互斥体（mutex）

使用范围

  • 资源地，数据库连接池 （我们希望看的是阻塞而不是失败）
    （blockQueue也可以）
  • 将任何一个容器变成有界阻塞容器

栅栏 Barrier

定义

  • 栅栏（Barrier类似闭锁），阻塞一组线程直到某事件发生。

与闭锁的区别

  • 所有线程必须同时到达栅栏位置才能继续执行
    • 几个家庭的 决定在某个地方集合 6点碰头，到了以后等其他人在讨论下一步
  • 闭锁是一次性对象，进入终止状态不能重置
  • 闭锁用于等待事件，栅栏用于等待其他线程

用途

  • 使一定数量的参与方 反复在栅栏汇聚（并行迭代十分有用）
  • 程序可以并行，但是需要等到某步骤的所有计算执行完毕进入下一个步骤。

CyclivBarrier过程

  • 线程到达栅栏位置时调用await,阻塞直到所有线程都到达栅栏
  • 所有线程到达了栅栏 ，栅栏打开，所有线程释放，栅栏重置下次使用
  • 如果await调用超时，或者await阻塞的线程中断，栅栏被认为打破，所有阻塞的await调用都终止抛出BrokenBarrierException
  • 成功通过栅栏，awit将为每个线程返回一个唯一的到达索引号
  • 利用索引“选举”一个领导线程，在下一次的迭代由领导线程执行一些特殊工作。

例子：细胞自动衍生

  • 构造函数根据CPU数量创建栅栏任务
  • 栅栏任务等所有board计算完 提交新值

另一种形式的栅栏 Exchanger

  • 在栅栏位置交换数据
  • 一个读缓冲区，一个写缓存区。用Exchanger回合，将满的缓冲区和空的缓冲区交换。（交换次数最低）
  • 通过Exchanger 安全发布

构建高效可伸缩的缓存Cache

缓存好处

  • 所有服务器应用程序都会采用，重用之前的计算结果，能有效降低延迟
  • 提高吞吐量，消耗更多内存

从HashMap出发，分析并发性缺陷

  • 用HashMap作为cache,但是HashMap补上线程安全，需要确保两个线程不会同时访问(但是只要一个线程能够执行compute)

第二种 ConcurrentHashMap （还可能有重复的计算（两个compute 一样）

  • 存在问题

  • 仍然可能的问题 （概率很小） conput 的if块 是非原子的
