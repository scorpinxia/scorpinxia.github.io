应用层协议原理

体系结构

  • Client-Server
  • P2P : BitTorrent 用户到用户的报文不需要通过中间服务器

进程通信

发起C 等待S Socket

可供应用程序使用的运输服务

传输层协议保证

  • 可靠数据传输：运输层协议提供 TCP 还有不可靠UDP（多媒体）
  • 吞吐量： 可用吞吐量就是发送进程能项接受进程交付bit的速率
  • 定时，发送注入套接字的每个bit到接收方不迟于100ms
  • 安全性

Intenet 运输服务

  • TCP
    • 面向连接 全双工 需要拆除连接
    • 可靠数据传输
    • 拥塞控制：不一定为通信进程带来好处，为整个Intenet带来 （限制每个TCP连接公平共享网络）

安全问题
TCP、UDP都无加密传入Socket的数据，可能被中间链路嗅探发现。
解决方法
TCP的加强版 （Secure Sockets Layer，SSL） （这种强化是在应用层实现的） （实际上是用SSl的API加密数据在传入TCP套接字、收端也是）

  • UDP

    • 轻量级 最小服务 无连接
    • 不可靠
    • 无拥塞控制 （任意速率往下网络层注入数据，实际吞吐量可能小于）

  • 不提供服务

    • 吞吐量和定时保证（自己设计，但是也不能保证）

注意的是许多防火墙设置阻挡UDP，所以SIP、Skype UDP通信失败就用TCP备份

WEB 和HTTP

HTTP

  • 两个程序：一个Client 一个Server，通过交换HTTP报文会话

  • Web 术语

    • Web page 由Object组成
    • Object：html jpeg java小程序 视频片段 且能通过URL寻址
    • html：多数Web有html文本；html文本通过URL地址引用页面其他对象
    • URL：例如http://www.someSchool.edu/someDepartment/picture.gif www.someSchool.edu主机名 someDepartment/picture.gif路径名
    • Web 浏览器：Firefox等 实现了Http的客户端
    • Web Server:实现了HTTP的服务端（存储所有web对象

  • HTTP

    • 定义了Webclient 向Webserver请求Web页面的方式、以及Server向client传送Web页面的方式
    • 使用TCP支持运输 ：Client给Server发送TCP连接，建立连接后，Client给Server发HTTP请求并接受HTTP响应报文。（HTTP不必担心报文丢失、恢复的细节，分层的好处）
    • Server给Clinet发送被请求的文件，不存储任何Client状态。比如客户几秒几次请求同个Object，Server会重新发送对象
    • 无状态协议（stateless protocol）

非持续连接和持续连接

单独TCP还是每次就一个，HTTP都支持，默认使用持续

非持续HTTP

  1. CLient 在80端口 发送给服务器www.someSchool.edu的TCP，Server Client 建立连接
  2. Client给Server HTTP请求报文。报文包含了路径名/someDepartment/home.index
  3. Server 结束，从Ram或磁盘检索出www.someSchool.edu/someDepartment/home.index ，在HTTP响应报文封装对象发送给Client
  4. Server进程通知TCP断开该TCP连接。（但是直到TCP确认客户完整收到响应，它才实际中断）
  5. Client 收到响应报文，报文指出封装的对象是一个html文件，客户从响应报文提取出该文件，检测该html文件，得到10个jepg图形引用
  6. 对每个引用的jpeg重复1~4

从上述看要，一共产生了11个连接。有些Jepg图形可能使用并行的TCP连接（大部分浏览器默认5~10）。用户可以设置最大并行连接数1，这样10个就会一条条串行建立

  • 往返时间：RoundTrip Time RTT ，请求到收到整个文件的时间

持续连接的HTTP

非持续连接的问题

  • 非持续连接给WebServer带来难题 （负担大）
  • 两倍RTT 交付时延 （创建和请求接受）

上面的11个对象通过一个连接传输,服务器发现连接时间间隔超过配置就关闭连接

HTTP报文

请求报文

分析报文

  • 普通的ASCII文本写的，容易读懂
  • 5行，每行换行回车结束，最后一行在加上换行回车
  • 请求行（一行或者更多），第一行。3个字段：方法字段、URL字段、HTTP版本字段。方法字段：GET POST HEAD PUT DELETE ，大部分是GET
  • 后续的行称首部行。
    • Host：www.someschool.edu(虽然已经在TCP连接了 ，但是这是web代理高速缓存所要求的)。
    • Connection：close ，该浏览器告诉Server不使用持续连接。
    • User-agent：浏览器类型
    • Accept-language:想得到该对象的发育版本（如果有的话，不然就发送默认的版本）

通用格式

实体体 entity body

  • POST方法使用
  • 提交表单常用，如搜索引擎关键词
    • 表单请求也常用GET,在请求的URL包含输入的数据 例如：一个表单使用GET 它有两个字段分别填monkeys 和bananas URL:www.someschool.edu/animalsearch? monkeys&bananas
  • 也可向服务器请求Web页面，取决表单字段输入的内容

HEAD 类似GET：当Server收到HEAD，用一个HTTP报文响应，但是不返回请求对象。（用来调试跟踪）

PUT:允许用户上传对象到指定的Server上指定的路径。也可被需要向Server上传对象的应用程序使用

DELETE:允许用户和应用程序删除Server上的对象

响应报文

分析报文

  • 初始状态行 status line
    • 3个字段 版本字段 状态码 相应的状态信息
  • 6个首部行
    • Connection：close ，该Server告诉Client发送完报文就关闭TCP连接
    • Date ：是发送该响应时报文的时候（不是检索 或者对象的修改的时间）
    • Server：提示什么服务器产生的 （类似user-agent）
    • Last-Modified：指示对象创建或者最后修改的时间日期（对对象缓存非常重要）
    • Content-Length：被发送对象的Bytes
    • Content-Type：指出entity body的对象类型
  • 实体体 entity body

Client 和Server的交互：cookie

起因

Web 站点无状态，但是Server希望识别用户，或限制用户访问，内容和用户身份联系，所以使用了cookie允许站点对用户跟踪

组件构成

  • HTTP响应报文的一个cookie首部行
  • HTTP请求报文的一个cookie首部行
  • Client端系统保留一个cookie文件，用户浏览器管理
  • Web站点的后端数据库

例子

假设Susan已经访问过eBay站点，它首次和Amazon。com联系，该Web站点产生一个唯一识别码作为索引在后端数据库产生一个表项。接下来Amazon web Server用一个包含set-cookoe的响应对Susan浏览器响应

浏览器收到报文，看到set-cookie在特定的cookie文件中添加一行，包含服务器主机名和对应的自己的识别码，sunan继续浏览Amazon网站，每请求一个Web Page 发往Amazon服务器的每个HTTP请求报文都包含

Web缓存（ proxy server）

示意图

请求对象www.someschool.edu/campus.gif

  • 浏览器创建一个到Web cache的TCP，向其发送HTTP请求
  • Web cache搜索本地 有就返回
  • 没有就打开一个与该对象初始服务器www.someschool.edu的TCP连接。 Web Cache在与Server的TCP连接上发送该对象的请求。
  • Web Cache得到该对象，在本地存储一个副本，并向Client发送该副本

通常由ISP购买安装，大学校园网安装一个多个缓存器，浏览器配置为指向它。

好处

  • 客户和初始Server的带宽远低于和Web cache的
  • 请求的东西在Web cache会很快
  • 大大减少一个机构接入到Intenet的通信量

原始网络

  • 流量强度： 假设1s15个请求 每个对象的平均长度1Mb
    • 局域网的流量强度
    0.15强度大概最多导致数10ms延时
    • 接入链路的流量强度

      1强度会让时延很大且无限增长

解决方法

  • 加链路速率（贵）
  • 加Web 缓存器

    假设命中率是0.4，流量强度就减小到0.6对应延时几十ms和Internet延时比就微不足道了

STMP

TCP底层 25端口

步骤

  1. Alice 调用邮件代理程序，提供BOB的邮件地址，书写并提示用户代理发送报文
  2. Alice的用户代理把报文发到了它的邮件服务器 ，报文在报文队列里
  3. Alice的有机物去的SMTP客户端发现了这个报文，创建到Bob邮件服务器的SMTP服务器的TCP
  4. 初始SMTP握手后，SMTP客户端通过TCP连接发送Alice报文
  5. Bob邮件服务器上 SMTP服务器接收改报文。Bob的邮件服务器将报文放入Bob邮箱
  6. Bob在之后可以用用户代理阅读

SMTP一般不使用中间服务器（就算离的特别远）

DNS

DNS提供的服务

识别主机的方式，主机名或者IP，人类喜欢主机名，路由器喜欢Ip。

Domain Name System DNS在UDP上 53号端口

DNS 的主要任务。

  • 分层的DNS server 实现分布式数据库
  • 主机能够查询分布式数据库的应用层协议
  • 其他协议（HTTP、SMTP、FTP）需求

通过主机得到Ip的步骤

步骤

  • 同一台主机运行着DNS应用的Client
  • 抽取URL的主机名 给DNS client
  • DNS Client给DNS Server发个包含主机名的请求
  • Client最终收到对应主机名的IP

其他服务

  • 主机别名 host aliasing 。规范主机名（
  • 邮件服务器别名
  • 负载分配：繁忙的站点被冗余分布在多台服务器上（不同IP）IP集合和同一个规范主机名联系。（会循环IP集合的次序分配）

DNS工作机理

Internet只用一个DNS服务器的设计可能有诸多问题

  • 单点故障
  • 通信容量
  • 远距离的集中式数据库
  • 维护（太庞大了）

设计

决定www.amazon.com的Ip

  • 先和根服务器之一联系
  • 返回顶级域名com的TLD（Top level domain) 服务器的ip，再联系
  • 返回amazon.com的权威服务器ip，在联系
  • 返回主机名ip

DNS服务器层次

  • 根DNS：400多个遍及全球，提供TLD服务器的ip
  • 顶级域DNS：com org net edu ,uk fr(国家相关的)，提供权威DNS服务器ip
  • 权威DNS： 可以自己建，可以买别人的记录

本地DNS的作用

  • ISP的提供的一个主机的DNS IP（通常通过DHCP）
  • 临近本主机
  • 交互步骤：假设TLD服务器 知道主机的权威DNS服务器IP
  • 交互步骤：TLD服务器一般只知道中间磨合DNS服务器，该服务器依次才能知道知道主机的权威DNS服务器IP （递归查询）

DNS缓存

  • 收到回答能映射缓存在本地存储器中，（就不需要权威服务器，也能提供 ）
  • 一般设置2天
  • 很容易绕过了根DNS

DNS 记录和报文

DNS服务器存储资源记录 Resource Record

  • [name,value,type,TTL]
    • TTL决定记录生存时间
  • type决定 name和Value
    • type=A name是主机名，value是主机对应ip
    • type=NS，name是个域（foo.com),value是个知道如何获得该域权威DNS的 主机名
    • type=CNAME，value是别名为Name的主机对应的规范主机名。
    • type=MX，value是别名为Name的邮件服务器对应的规范主机名。

权威服务器（包含该主机）和非权威服务器的区别

  • 权威服务器只有一条包含类型A的记录
  • 非权威服务器

DNS报文

查询和回答一致

  • 前12字节的首部
    • 标识符 16bit 标识查询
    • 标志 查询、回答？权威？递归？
    • 四个后面的数量
  • 问题区域
    • 查询的信息
    • 名字字段
    • 类型字段 A、MX等
  • 回答
    • 包含RR
    • 可以多个（一个主机多IP）
  • 权威区域
    • 其他权威服务器的记录
  • 附加区域
    • 有帮助的记录
    • 比如MX请求回答的规范主机名，附加区域会有规范主机名的A记录
