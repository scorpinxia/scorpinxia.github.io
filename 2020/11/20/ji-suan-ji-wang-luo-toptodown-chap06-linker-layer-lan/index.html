交换局域网

链路层寻址和ARP

MAC地址

为什么要有链路层地址

  • 不是主机和路由器有链路层地址，而是他们的适配器（网络接口）具有链路层地址。多个网络接口就有多个链路层地址（相关联的多个IP一样）
  • 分层原理 局域网适用于任意网络层 不一定是IP

链路层任务
在主机和路由器直接承载数据报。

链路层地址称呼 48bit 唯一

  • LAN 地址
  • 物理 地址
  • MAC 地址（最常用）

适配器向目的适配器发送帧

  • （插入目的的MAC，发送到局域网）
  • 有的需要局域网内所有其他适配器处理 MAC广播地址FF-FF-FF-FF-FF-FF

ARP

假设简单的一个主机1Ip和1MAC,交换机广播帧

  • 过程
    • 222.222.222.220向222.222.222.222发IP数据报。他们在相同的子网里面。
    • 为了发送需要向适配器提供IP数据报和目的222.222.222.222的MAC地址
    • ARP在发送主机的ARP模块（局域网内然后IP输入就能得到AMC）
    • 当ARP表没有的时候，发送方构造一个特殊的ARP Packet。（包括 发送、接受IP 接受MAC字段） 广播

      ARP是封装在链路层的，也包含网络层地址（可以是跨越链路层和网络层的协议）不一定在哪个层

发送数据到子网以外

过程

  • 111.111.111.111想给主机222.222.222.222发送数据报，需要给适配器指示MAC
  • 这个MAC是111.111.111.110 路由器接口的MAC。通往最终目的的第一跳路由器IP.
  • 路由器得到这个数据 查表继续向222.222.222.220转发

以太网

帧结构

  • 数据字段 46~1500bytes
  • 目的地址
  • 源地址
  • 类型字段：允许以太网复用多种网络层协议
  • CRC:4bytes
  • 前同步码：前7byte 10101010 最后一个字节10101011 唤醒接收适配器，他们时钟和发送方时钟同步。 以太网类型 10Mbs\100Mbs\1Gps

链路层交换机

交换机的转发和过滤

  • 过滤：决定一个帧被转发还是丢弃

  • 转发：决定一个帧被导向哪个接口

  • 交换机表：

    • 一个MAC
    • 一个通往MAC的接口
    • 表项时间

  • 交换机表的三种情况 MAC DD-DD-DD-DD-DD-DD从x接口到达

    • 没有表项联系该MAC：向除x接口外所有接口前面的输出缓存转发该帧副本（相当于广播）
    • 有个表项把该MAC和接口x联系起来，无须转发任何接口，直接丢弃
    • 有个表项把该MAC和接口y!=x联系起来.转发对应接口就可

  • 现代分组交换机可以被配置（IP或者MAC转发）

交换机的优点

  • 即插即用（不用网络管理员）

交换机的自学习

学习MAC到达的接口和时间（设置老化期）

交换机毒化

  • 通过伪造表项填满交换机表

交换机和路由器比较

不同层级

交换机的优点、缺点

  • 优点
  • 第二层
    • 即插即用
    • 高分组过滤和转发速率
  • 缺点
    • 对广播风暴没有任何保护（如果主机一直广播以太网帧，交换机会一直转发，导致整个以太网崩溃）

优缺点对比

组网意见

  • 几百台主机 小网络几个局域网网段,交换机就够了，不要求IP地址的然后配置就能使流量局部化。
  • 几千台，网络中（除了交换机）还需包括路由器，对广播风暴控制

VLAN

交换局域网的问题

  • 缺乏流量隔离。有些广播帧跨域了 整个机构网络
  • 交换机的无效使用。单个96端口的不能隔离10个组
  • 管理用户。改变物理布线，才能连到不同交换机

VLAN的作用

  • 单一物理局域网定义多个虚拟局域网

VALN的困难

  • 完全隔绝的两个VLAN的，流量怎么交换？
    • 通过VLAN的一个端口连接外部路由（属于两个VLAN的）

VLAN trunking

特殊的端口被配置为干线trunking，发送到任何VLAN的帧，通过干线链路转发到其他交换机

扩展的以太网帧 802.1q VLAN

  • VLAN还能基于IP 和MAC

回顾Web页面请求过程（From top to Down）

假设情况

  • 一个学生用便携机和学校的以太网交换机相连，交换机又与学校的路由器相连。
  • 学校的路由器和一个ISP相连（Comcast.net)
  • 这里comcast.net为学校提供DNS服务。DNS服务器在Comcast网络，而不是学校网络
  • DHCP服务器在路由器中

准备1、DHCP、UDP、IP 以太网

过程

  • 首先网络连接，没有IP就不能做任何事情。所以Bob的便携机运行DHCP协议从本地获得一个IP以及其他信息
    • Bob操作系统生成DHCP request 报文 ，放入目的端口67（DHCP server） 和源端口68（DHCP Client）的UDP报文段
    • 报文段被放置在一个广播IP目的地址255.255.255.255，和源IP0.0.0.0（还无IP）的IP数据报中
    • IP数据报被放入以太网帧中 目的MAC FF:FF:FF:FF:FF:FF ,源MAC 就是自己的MAC:XXXX。使该帧广播到与交换机连接的所有设备（如果顺利 包含了DHCP服务器）
    • DHCP请求的广播以太网帧是第一个Bob发送到以太网交换机的帧。所以该交换机在所有的出端口广播入帧，包含连接路由器的端口
    • 路由器在它具有MAC的XXXXXX 接口收到该广播以太网帧，该帧包含DHCP请求，并且从该以太网帧抽取IP数据报。该数据报的广播IP目的地址指示其需要高层协议处理。数据报的payload（UDP报文）分解，向上到达UDP。DHCP服务器由来DHCP请求报文。
    • 假设DHCP可分配 68.85.2.0/24 块。DHCP服务器生成了包含IP、默认网关、子网掩码的DHCP ACK报文。一直下放到以太网帧源MAC 路由器的MAC 目的也就是Bob的MAC
    • 包含DHCP ACK的以太网帧从路由器发到交换机。交换机是自学习的，先前知道了Bob的位置，就只往Bob的输出端口转发。
    • Bob 接收到DHCP ACK以太网帧。得到UDP报文段，得到DHCP ACK报文。得到Ip和DNS serverIP IP转发表安装默认网关，除了 子网内的地址都会往 默认网关发送目的地址
    • *（DHCP后只有两个步骤是实际必要的）**

准备2 DNS和ARP

Bob的Web浏览器生成一个TCPSocket用于给www.google.com发送HTTP请求。为了生成这个套接字需要知道www.google.com的IP（TCP Socket 需要四元组） 所以需要DNS

  • Bob的操作系统生成了DNS查询报文，把字符串www.google.com放入DNS报文的问题段中。DNS报文被放置在具有53端口号（DNS服务器）目的端口的UDP报文段。该UDP报文段则放入（DHCP ACK的 DNS服务器IP）和源IP BobIp的IP数据报里面
  • 数据报放入以太网帧。在链路层寻址网关路由器。虽然知道网关路由器的IP，但是不知道网关路由器的MAC，为了获得网关路由器的MAC，需要使用ARP
  • Bob生成了一个具有目的IP地址的（默认网关）的ARP查询报文，放入具有广播目的地址（FF:FF:FF:FF:FF:FF)的以太网帧，向交换机发送。交换机把该帧交付给所有连接的设备（包括网关路由器）。
  • 网关路由器的接口收到了包含该ARp查询的报文的帧，发现ARP报文的目标IP对应接口IP。网关因此准备了一个MAC回答，指示自己的对应IP的MAC，放入以太网帧，目的地址是Bob的MAC地址。
  • Bob收到ARP回答报文的帧，从报文得到网关的MAAC
  • Bob可以使包含DNS查询的以太网帧寻址到网关路由器的MAC。注意 该帧的目的地址网关路由器MAC,该帧的数据报目的地址IP是DNS服务器IP。Bob给交换机发送该帧，交换机将该帧交付到网关路由器。

准备3： 域内路由选择到DNS服务器

  • 网关路由器收到该帧抽取包含DNS查询的IP数据报。查找该数据报的目的地址（DNS服务器IP），根据转发表决定发到Comcast网络最左边的路由器，IP数据报放到链路层分发到该路由器
  • Comcast 最左边的路由器接收到帧，抽取IP数据报，查找目的地址（DNS IP）根据转发表确定接口，（转发表有 Comcast域内协议 RIP、或、OSPF、IS-IS等域内协议和Internet的域间协议BGP填写）
  • 包含DNS查询的IP数据报到达了DNS服务器，服务器抽取出DNS查询报文，在它的DNS数据库 中查找名字www.google.com IP的源记录（来源google.com的权威服务器），DNS服务器形成包含主机名到IP地址映射的DNS回答报文。放入UDP报文段，放入到BobIP的数据报。
  • Bob得到了服务器www.google.com的IP。

Web Client - Server 交互：TCP HTTP

  • Bob得到了服务器www.google.com的IP，来生成TCPSocket。生成TCP socket时必须对www.google.com的TCP三次握手。Bob先生成了目的端口80（HTTP） TCP SYN报文段，放入目的IP （网站IP） 的IP数据报，放入MAC地址（网关路由器MAC）的帧中，发送给交换机。
  • 经过多个路由转发，最终包含TCP SYN报文到达www.google.com。数据报抽取出来并分解到和端口80联系的欢迎套接字。生成一个连接套接字（根据收到的四元组）。然后产生一个TCP SYN ACK报文段，放入数据报（目的Bob）放入链路层帧，往回传。
  • Bob得到了数据报，被操作系统分解到之前已经生成的TCP Socket的里面，进入连接状态。
  • 现在可以通过这个TCP socket，Bob浏览器生成包含要获取URL的HTTP GET报文。交付到报文段、数据报 最后到www.google.com。
  • www.google.com 的HTTP服务器从TCP socket 读取HTTP GET报文，生成HTTP 响应 返回
  • Bob 的Web流量从套接字读取到HTTP响应，抽取出html,就显示了。
