Java-concurrency-in-pratice-Part03-Active、efficiency、Test-chap10-ActiveDanger

笔记 自学
JAVA Concurrency
发布日期:   2020-12-30

DeadLock

JVM不像数据库 ，无法自己解决死锁

  • 并不是一定发生，死锁往往发生在高负载

简单顺序死锁

例子

解决方法

  • 如果所有线程以固定的顺序获得锁，程序中就不会出现锁顺序死锁。

动态的锁顺序死锁

例子1 锁住账户原子转账

  • 当X向Y Y向X可能死锁

解决办法 System.identityHashCode 定义顺序

  • 可能Hashcode 相等，但是如果Account有唯一不可变键值

死锁的问题 (几十亿操作一次，也会出大问题)

  • 短时间持有锁，看似降低竞争难度，增加了测试中找出潜在死锁的风险

例子2 相互协作对象

  • setLocation 和getImage都会尝试获得2个锁
    • GPS接收器线程调用set 更新位置，判断是否到达，通知Dispatcher需要新的目的地 先尝试taxi的锁，在Dispatcher锁
    • getImage 先Dispatcher锁，在获取每个Taxi锁
  • 持有锁调用某个外部方法，可能会获取其他锁

开放调用 （Open Call）

定义

  • 调用某个方法不需要持有锁（类似封装提供线程安全）
  • 同步代码块仅用于保护涉及共享状态的操作

公开调用避免死锁

  • 丢失了一点原子性。（位置快照）
  • 更新出租车位置和通知调度程序这辆出租车去新地方（可以不是原子操作，）

资源死锁

定义例子

  • 两个不同数据库连接池，资源池通常用信号量实现，请求两个资源不会遵循相同顺序，线程A D1数据库 线程B D2数据库
  • 线程饥饿死锁：一个任务提交另一个任务，等待另一个人任务完成，（任务等待其他任务结果）

死锁避免和诊断

程序编写总体策略

  • 一个程序每次至多一个锁（就无锁顺序死锁）（不太现实）
  • 减少潜在加锁交互，获取锁需要遵循 的协议写入正式文档
  • 尽可能用开放调用

程序检查总体策略

  • 找出什么地方多个锁（代码块）
  • 确保他们整个程序获取锁顺序一致
  • 开放调用情况下（可以自动化分析）

支持定时的锁

显示Lock类

  • 可以设置Timeout 等待超过就返回失败

线程转储信息分析死锁（Thread Dump)

JVM 功能（许多IDE支持）

  • 三个组件 只有交互才会显现出来
  • 例子：不同调用链通过JDBC驱动以不同顺序获取多个锁，第二个问题：多个线程试图使用同一个JDBC连接。（因为JDBC规范并无要求Connection 线程安全，以及Connection是封闭在单个线程）

其他活跃性危险

饥饿

定义

  • 线程无法访问它需要的资源（优先级不当，或者无法结束）
    （CPU时钟之类的）

    建议

  • 大多数并发程序，尽量不用改变优先级（这依赖平台）

糟糕的响应性

例子

  • GUI程序使用了后台线程，CPU密集型 和事件线程一起竞争CPU
  • 某线程长时间占用锁

活锁

定义

  • 不阻塞线程，但不断重复执行相同操作，且总会失败

例子

  • 处理过渡的错误代码，错误的将不可修复的错误作为可修复的错误。
  • 两台机器同样的载波发数据报冲突，稍后再发，又冲突，不断冲突(因为用相同的载波改变机制)

解决办法

  • 重试机制引入随机性。
  • 以太网协议 定义在重复发生冲突采用指数回退机制
  • 并发应用 等待随机长度时间 回退避免死锁
