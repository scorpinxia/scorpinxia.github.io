任务取消

java不能安全抢占停止线程，所以只能协作机制

  • 标志取消法，让素数生成器运行1s后取消。通常不会刚好运行1s停止 可能刚好在循环下一次

中断

在素数退出的时候，阻塞put

  • 生产者速度超过消费者处理速度
  • 队列填满 生产者put阻塞
  • 消费者希望取消生产者任务，cancel设置标志
  • 生产者永远检查不了标志（在阻塞put)

中断的作用
每个线程都有一个bool中断状态

  • interrupt 中断线程
  • isinterrupted 返回状态
  • 静态的 interrupted 清除中断状态返回之前的状态

何时检查中断
Thread.sleep 和Object.wait 都会检查线程何时中断，发现中断返回

上述阻塞库方法 响应中断的操作

  • 清除中断状态
  • 抛出InterruptedException

线程非阻塞状态中断 中断状态设置

中断操作的理解

  • 不会中断正在运行的线程，只是发出中断请求，在何时时候中断自己
  • 中断式实现取消的最合理方法

响应中断

可中断的阻塞的两种响应 例如Thread.sleep BlockingQueue.put

  • 传递异常，从而你的方法也可中断的阻塞

  • 恢复中断状态，从而调用栈上层代码处理
    不是在捕获的时候（可能无限循环）

例子：计时运行（有些问题永远没有结果）

找出10分钟的的答案

  • 指定时间运行任意一个Runnable实例

    中断线程之前，无法知道调用线程的中断策略
  • 专门线程中断任务


    存在join（等待线程死亡) 无法知道执行控制是因为线程正常退出还是jion超时

Future 来取消

Future 抽象机制管理任务生命周期，处理异常、取消。

处理不可中断的阻塞

等待获得内置锁的阻塞或同步IO的阻塞，中断请求只能设置线程中断状态，我们要指导阻塞原因

IO阻塞

java.io同步Socket I/O 常见的阻塞I/O形式：对套接字读取写入。

  • 解决方法，虽然题目不响应中断，关机底层的套接字，可以使执行read、wirite方法阻塞的线程抛出 SocketException

java.io同步 I/O ；中断一个正在InterruptibleChannel上等待的线程时，将抛出ClosedByInterruptException ，并导致所有链路上操作阻塞线程抛出ClosedByInterruptException。大多数标准Channel都实现了InterruptibleChannel。

java.nio 调用Selector.select阻塞了 close 抛出closedselectorException

获取某个锁

线程认为自己肯定会获得锁，所以不好中断。 Lock类有lockinterruptibly 允许等待锁同时响应中断

停止基于线程的服务

背景

  • 应用程序有多个线程的服务，例如线程池，服务生命周期比创建它们方法生命周期更长
  • 应用程序停 服务线程也需要停 服务拥有的线程也需要停止

封装原则：除非拥有某个线程，否则不能对线程控制 （中断 或者修改线程优先级）。
线程所有权不可传递: 应用程序有服务 服务有工作者线程 应用程序不拥有工作者线程，不能直接停止工作者线程。
例子

  • ExectorService 提供了shutdown 和ShutdownNow 方法（关闭自己和它拥有的线程）。其他拥有服务的也需要类似机制

日志服务

大多数服务器应用程序用到日志，print。PrintWriter时线程安全的，但是高容量应用程序需要一定的开销。log方法将日志消息放入某个对了，由其他线程来处理

  • 多消费者单生产者 （缺少停止）

直接使日志线程退出的问题 （由于反复take take能响应中断，直接捕获中断退出）

  • 丢失其他等待写入消息
  • 其他线程调用log阻塞 （消息队列满），这些线程无法解除阻塞

设置“请求关闭”标志，避免进一步提交日志消息。然后把队列所有消息写入，并解除log阻塞生产者

  • 存在问题
    • 竞态条件：先判断在运行（可通过宣布清空等待数秒，降低概率，没解决实际问题）
  • 解决方法
    • 日志提交操作原子化
    • 原子方式检测关闭，有条件递增保持提交

关闭ExecutorService

  • shutdownnow()立即所有
  • shutdown 等待队列所有任务执行完

“毒丸”对象 关闭服务

当生产者提交“毒丸”对象，不会在提交任何工作，而FIFO队列消费者提交毒丸之前的工作都会被处理

  • 问题只有消费者 生产者数量已知情况可以用“毒丸”

    桌面搜索：单生产者-单消费者例子


扩展多个生产者：每个生产者向队列放一个“毒丸”，当消费者收到对应数目“毒丸”：才停止。

  • 缺点
    • 只能无界队列
    • 太多数量 难使用

例子：只执行一次的服务

shutdownNow的局限性

shutdown，会尝试取消正在执行的任务并返回所有已提交但未开始的任务（可以写入日志或保存起来 以便以后进行处理）

处理非正常终止线程

单线程运行 未捕获异常 终止，程序停止运行，输出栈信息
多线程运行 某线程出故障，控制台信息

  • 问题： RuntimeException等通常不会被捕获

任务处理线程 （线程池的工作者 Swing的事件派发线程），将通过某种抽象机制（Runnable 等)运行未知代码

调用每个方法都要持怀疑态度

解决方法

  • try catch finally

未捕获异常的处理

  • Thread API:Uncatught-ExceptionHandler

步骤

  • 线程由于未捕获异常退出时，JVM会把这个事件报告给应用程序提供的异常处理器
  • 没有任何异常处理器 默认将栈追踪信息输出到System.err

例子

JVM关闭

方式

  • 正常关闭：最后一个正常线程结束，或者调用了System.exit Ctrl+C（SIGINT)
  • 强制关闭：Runtime.halt 或者操作系统”杀死“ JVM进程
    (SIGKILL)

关闭钩子

  • 正常关闭：
    • 调用所有已注册的关闭钩子（shutdown Hook)（通过Runtime.addShutdownHook注册的没开启的线程）。
    • 关闭了应用程序线程，如果有其他（守护 非守护）线程还在运行，将于关闭进程并发。
    • 所有关闭钩子执行结束 如果runFinalizerOnExit true，那么JVM运行终结器 停止
    • 如果关闭钩子 或终结器没完成，那么 正常关闭进程“挂起” JVM被强制关闭，只关闭JVM，不运行关闭钩子

关闭钩子的作用

  • 实现服务应用程序的清理 （清理临时文件，无法由操作系统自动清除的资源）

关闭钩子注意点

  • 关闭钩子并发 ，不应该依赖可能被应用程序和其他关闭钩子关闭的服务
  • 对所有服务同一个关闭钩子（不是每个服务一个）

守护线程

作用

  • 创建一个线程执行辅助工作，但不希望线程阻碍JVM关闭（Daemon Thread)

线程分为守护线程和普通线程

  • JVM启动创建的线程，除了主线程都是守护线程（垃圾回收机和其他辅助工作线程）
  • 创建一个新线程，新线程继承创建它的线程守护。所以默认主线程创建所有线程为普通线程

守护线程和普通线程的区别

  • 线程退出发生的操作，一个线程退出，JVM检查其他在运行的线程，如果这些线程都是守护线程，JVM会正常退出操作。JVM停止时，所有仍然存在的守护线程将被抛弃（既不finally 也不会追踪栈，JVM直接退出
  • 尽可能少使用守护线程
    • 包含I/O的守护 很危险（不清理 直接抛弃）
    • 最好是“内部”任务，周期移除内存缓存逾期数据

终结器

不需要内存资源，可通过垃圾回收期回收他们，其他资源（文件句柄或者套接字句柄），需要显式交还给操作系统。

  • 通过垃圾回收机对finalize 方法对象特殊处理完成

避免使用包含终结器的类（通过finally 和显式close 更好）
