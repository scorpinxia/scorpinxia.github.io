Base

URL URI

  • URL（Locator) : web 服务器 需要输入网址
    http://hacker.jp
  • URI (identifier):某一互联网资源,URL是URI子集
    • ftp://xxxx/rfc/rfc1808.txt
    • telnet://192.0.02.16:80/
    • http://www.xxx.org/rfc/rfs.txt

      URI格式

指定URI，需要使用涵盖全部必要信息的绝对URI、绝对URL、相对URL。

  • 相对URL，基本 URI指定的URL /image/logo.gif
  • 绝对URI
    • 登录信息：可选
    • 服务器地址：可以是hacker.jp（DNS可解)、ipv4、ip
      v6
    • 服务器端口：可选 或默认
    • 服务器文件路径 。UINX目录
    • 查询字符串：可选，针对已指定文件路径资源，使用查询字符串传入参数
    • 片段标识符：可选，已获取资源的子资源（RFC无明确规定使用方法）

简单HTTP

get:获取资源

post：传输实体主体

虽然get 也可以，但一般用post

put: 传输文件

  • HTTP/1.1 put方法自身不带验证，一般WEB网站不使用
  • 配合WEB应用程序的验证机制，或者架构设计采用REST(REpresentational State Transfer 表征状态转移) 标准的同类 Web可能开放

head: 获得报文首部

  • 与get一样，只是不返回报文主体部分。用来确认URI有效性及资源更新等

delete:删除文件

  • 与put一样的使用情况

options：询问支持的方法

trace：追踪路径

通常不用

connect：要求用隧道协议连接代理

主要使用 SSL(Secure Sockerts Layer ) TCS (Transport Layer Security 传输层安全)协议把通信内容加密 通过网络隧道传输

通信数据转发程序： 代理、网关、隧道

代理

理由

  • 缓存技术减少带宽

  • 组织内部对特定网站的访问控制，获取访问日志

  • 缓存代理

    • 转发响应，预先缓存资源在代理服务器。
    • 再次受到相同资源请求，不从源服务器获取

  • 透明代理

    • 不对报文进行任何加工

  • 非透明代理

    • 反之

网关

  • 提高通信安全，网关可以连数据库用SQL查询，与其他系统联动等

隧道

  • 建立与其他服务器的通信链路（比如SSL）确保客户端与服务器的安全通信。
  • 隧道不解析HTTP，原样转发

Web安全的https

HTTP 缺点

通信明文 可能被窃听

解决办法
HTTP+SSL

不验证身份 可能伪装

  • 无限定 发送IP 端口情况下

可能结果

  • 无法确定请求发送到目标是否真实返回（可能伪装的Web)
  • 无法确定返回到的客户端是（可能是伪装的客户端）
  • 无法确定正在通信的对方是否有访问权限
  • 无法判定请求来着哪
  • 无法阻止海量请求的Dos (Denial of Service)

解决办法 （SSL提供的证书）

  • 查明对手的证书 （客户端持有证书 也可以完成个人身份认证）

报文完整性 无法保证（可能篡改）

之前的方法 数字签名 MD5散列值 （需要操纵客户端自己检查验证且同样可以改写）。

HTTP+加密+认证+完整性保护=HTTPS

HTTPS=HTTP+SSL

通信接口部分用SSL和TLS协议替代

  • 先公开传 共享 ，在用共享

证明公开密钥正确性的证书

为了证明公开密钥本身是货真价实的公开密钥

  • 比如和某台服务器建立公开密钥加密通信，证明证明收到的公开密钥是那台服务器的公开密钥

  • CA Certificate Authority

    • 公钥证书给客户端（数字证书）

企业、组织认证（EV SSL 证书）

客户端证书

  • 一般比较少
  • 银行的网上银行需要（确认特点终端身份）

证书也可以伪造（被吊销需要时间）

  • OpenSSL 可构建自己认证机构给自己服务器证书，互联网上无用

HTTP的安全通信机制

图

  1. 发送Client Hello 开始SSL通信（报文包含SSL版本、加密组件、算法）
  2. SSL Server Hello 回答 （报文包含根据客户端筛选的SSL版本、加密组件、算法）
  3. 服务器发送Certificate 报文。（公共密钥证书）
  4. Server Hello Done 报文 客户端 最初的SSL握手协商部分结束
  5. 客户端 Client Key Exchange 报文 。随机密码串 Pre-master secret ，报文用公开密钥加密
  6. 客户端 Change Cipher Spec 报文。提示服务器报文通信采用 Pre-master secret
  7. Finished 报文 。包含全部报文的整体校验值，握手协商的成功决定因素（能否正确解密改报文）
  8. 服务器一样的Change Cipher Spec
  9. Finished
  10. 服务器、客户端都Finished报文交换完，SSL连接建立完成。在开始应用层协议的通信 HTTP请求
  11. HTTP响应
  12. 最后由客户端断开连接，close_notify报文

在上述中，应用层会发送数据附加MAC(Message Authentication Code) ，查知报文是否篡改。（完整性）

基于HTTP的协议

HTTP瓶颈（如何把内容及时反馈到客户端）

问题

  • 一连接 一请求
  • 请求只能从客户端开始。客户端不可以接受响应外指令
  • 请求、响应首部未经压缩发送。（首部信息多延迟就大）
  • 发送冗长的首部。（每次相同首部浪费）
  • 可选择任意压缩数据格式。非强制

Ajax解决办法（异步JavaScript和XML） （解决实时性）

  • 有效利用JavaScript 和DOM（Document Object Model 文档对象模型） ，局部Web页面替换加载的异步通信手段
  • 核心 XMLHttpRequest的API 通过JavaScript 脚本语言调用和服务器进行HTTP通信

Comet 的解决方法

SPDY

新加会话层

SPDY的功能

  • 多路复用流（单一TCP 无限制处理多个HTTP请求）
  • SPDY 无限制并发处理，并给请求逐个分配优先级
  • 压缩HTTP首部
  • 推送功能（服务器直接发生，不必等待客户端请求）
  • 服务器提示功能： 主动提示客户端请求所需资源

SPDY发展

  • 实际的Web网站进展不佳
    • 只是将单个域名（ip）的通信多路复用，一个Web网站使用多个域名的资源，改善效果有限制
  • SPDY是有效消除HTTP瓶颈，而很多Web网站的瓶颈不是HTTP

使用浏览器进行全双工通信的WebSocket

背景

  • 利用Ajax和Comet 进行通信可以提升Web的浏览速度，通信采用HTTP就无法彻底解决瓶颈问题。

WebSocket 的设计和功能

目的

  • 解决Ajax和Comet的XMLHttpRequest附带的缺陷所引起的问题

协议特点

  • 建立起WebSocket协议后，通信可互相发送JSON、HTML、任意格式数据
  • 建立在HTTP基础上的协议，发起仍是客户端

协议目的

  • 推送功能
    • 服务器直接发送数据，不必请求
  • 减少通信量
    • 一直保持连接不需多个，WebSocket首部信息也小
    • 在WebSocket之前需要握手步骤
      • Key 代表独一的键值
    • 握手响应

      • Aceept由 之前的Key生成

HTTP首部注入攻击

造成影响

  • 设置任意Cookie
  • 重定向任意URL
  • 显示任意的主体（HTTP Respnse Splitting Attack）

案例

HTTP 响应截断攻击
